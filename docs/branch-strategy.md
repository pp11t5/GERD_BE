# 브랜치 전략

## 1. 문서 목적
이 문서는 팀 브랜치 운영 전략과 PR/코드리뷰 방식을 정리한다.

## 2. 기본 브랜치 구조

| 브랜치 | 역할 |
|---|---|
| `main` | 배포 기준 브랜치. 직접 push 금지, PR 병합으로만 반영 |
| `develop` | 통합 개발 브랜치 |
| `feature/*` | 기능 단위 개발 브랜치 |
| `fix/*` | 버그 수정 브랜치 |
| `init/*` | 프로젝트 초기 설정 브랜치 |

## 3. PR 규칙

- `main` 브랜치는 직접 push 금지
- 리뷰어 1명 이상 승인 후 병합
- CI 통과 필수 (빌드 + 테스트)

## 4. CI 통과 기준

PR 병합 전 아래 조건을 모두 충족해야 한다.

| 항목 | 기준 |
|---|---|
| 빌드 | `./gradlew build -x test` 성공 |
| 테스트 | `./gradlew test` 전체 통과 |
| 대상 브랜치 | `main`, `develop` 대상 PR |

- 테스트 실패 시 병합 불가 (GitHub Branch Protection Rule로 강제)
- CI는 `.github/workflows/ci.yml` 에서 실행

## 5. Git 커밋 컨벤션

커밋 메시지는 아래 형식을 기본으로 한다.

`타입: #이슈번호 작업 내용`

예시:

- `feat: #12 회원가입 API 구현`
- `feat: #1 프로젝트 초기 세팅`
- `fix: #27 로그인 예외 처리 수정`
- `refactor: #34 출석 서비스 로직 분리`
- `test: #12 회원가입 서비스 테스트 추가`
- `docs: #8 브랜치 전략 문서 보완`
- `chore: #3 GitHub Actions 워크플로우 수정`

타입은 아래 기준으로 사용한다.

| 타입 | 의미 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변화 없는 구조 개선 |
| `test` | 테스트 코드 추가/수정 |
| `docs` | 문서 작성/수정 |
| `chore` | 빌드, 설정, 의존성, CI 등 기타 작업 |

작성 규칙:

- `타입`은 영문 소문자로 작성한다
- `#이슈번호`는 이슈 트래커의 번호를 사용한다
- 작업 내용은 한눈에 변경 사항을 알 수 있게 구체적으로 작성한다
- 하나의 커밋에는 하나의 의도를 담는다
