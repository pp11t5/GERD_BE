# Performance Check

## Purpose

이 스킬은 백엔드 코드/구조를 성능 관점으로 분석할 때 사용한다.
이론적인 최적화 나열보다 실제 트래픽 기준의 병목 근거, 개선 우선순위, 정량적 기대 효과를 우선한다.

## Critical Rules

- 추측 최적화보다 병목 근거를 우선한다.
- 항상 실제 트래픽 기준으로 설명한다.
- 항상 우선순위를 요구한다.
- 항상 정량적 결과를 요구한다.
- DB 쿼리 수, 응답 크기, 외부 호출 횟수, 캐시 적중률을 함께 본다.
- 캐시는 정합성과 무효화 정책이 설명 가능할 때만 제안한다.
- 실행계획, p95, TPS, latency, error rate 같은 측정 지표를 가능한 한 명시한다.
- 측정값이 없으면 무엇을 추가 수집해야 하는지 먼저 정리한다.

## When To Use

- 구현 전 성능 리스크를 미리 점검할 때
- 리뷰 중 병목 가능성과 우선순위를 정리할 때
- JPA/QueryDSL 조회 성능을 분석할 때
- API 응답 속도 개선안을 구조적으로 받고 싶을 때
- Redis 캐싱 전략이나 K6 부하 테스트 시나리오가 필요할 때
- 포트폴리오/면접용으로 실무형 성능 개선 문서를 만들 때

## Analysis Focus

1. 전체 구조 병목: CPU, 메모리, DB, 네트워크
2. DB / JPA: N+1, 인덱스, join, fetch 전략, 실행계획
3. API 응답 속도: Controller -> Service -> DB -> 외부 API
4. 캐싱 전략: 캐시 대상, key, TTL, 무효화, Redis 자료구조
5. 부하 테스트: 사용자 행동 기반 시나리오, p95, error rate
6. 개선안 정리: 단기 개선과 중장기 구조 개선 분리

## Required Angles

- 정량 지표를 포함한다.
  - 예: `p95`, `TPS`, `latency`, `쿼리 실행 횟수`, `캐시 적중률`
- 우선순위를 포함한다.
  - 예: `즉시 적용`, `단기`, `중장기`
- 실제 트래픽 기준으로 설명한다.
  - 예: 피크 시간대, 동시 사용자 수, 요청량
- 재측정 계획을 포함한다.
  - 예: 개선 후 어떤 지표를 어떤 조건으로 다시 볼지

## Input Guide

프롬프트에 아래 입력을 많이 줄수록 결과 품질이 좋아진다.

- 코드 또는 주요 메서드
- 엔드포인트 설명
- 예상 트래픽
- DB 스키마/인덱스
- 실제 느린 쿼리 또는 로그
- 외부 API 호출 유무
- Redis 사용 여부
- 목표 지표
- 병목이 의심되는 실제 구간

## Review Checklist

- N+1 가능성이 있는가?
- 불필요한 전체 조회/전체 카운트가 있는가?
- 응답 DTO가 과도하게 무거운가?
- 외부 API 호출이 트랜잭션 안에 있는가?
- 반복 계산/중복 조회를 줄일 수 있는가?
- 인덱스 없는 검색/정렬 조건이 있는가?
- 페이징, 배치, 캐시 전략이 필요한가?
- 커서/정렬 쿼리에서 필터 컬럼과 정렬 키를 함께 고려한 인덱스가 필요한가?
- 서비스 레벨 중복 체크만으로 끝내지 말고 unique constraint가 필요한가?
- 요청 처리 흐름별 병목을 나눠 설명하는가?
- 캐시를 제안하면 key, TTL, 무효화까지 설명하는가?
- 부하 테스트를 말하면 p95, error rate, 재측정 계획이 포함되는가?

## Workflow

1. 분석 대상을 코드, 쿼리, API, 캐시, 부하 테스트 중에서 먼저 고른다.
2. 병목 후보를 DB, 애플리케이션, 외부 연동, 캐시 계층으로 나눈다.
3. 실제 트래픽과 측정 지표 기준으로 어떤 구간이 먼저 터질지 본다.
4. 빠른 개선과 구조적 개선을 분리해서 우선순위를 세운다.
5. 개선 후 어떤 방식으로 재측정할지 함께 남긴다.

## Completion Gate

- 병목 지점이 단순 추측이 아니라 근거 기반으로 설명되는가?
- 우선순위와 정량적 기대 효과가 포함되는가?
- 실제 트래픽 기준 설명이 들어가는가?
- 개선 후 재측정 계획이 있는가?
