# Scheduler Strategy

## Purpose

이 스킬은 현재 프로젝트에서 Spring Scheduler를 도입하거나 수정할 때 사용한다.  
스케줄러는 편한 기능이지만, 중복 실행, 장애 전파, 운영 가시성 부족 문제가 쉽게 생기므로 명확한 기준 아래에서만 사용한다.

## Critical Rules

- 스케줄러는 “정해진 주기로 반복 실행되는 백그라운드 작업”일 때만 사용한다.
- 사용자 요청-응답 흐름을 대체하는 용도로 스케줄러를 쓰지 않는다.
- 스케줄러 메서드 안에 모든 비즈니스 로직을 넣지 말고, 전용 서비스 메서드를 호출하게 한다.
- 중복 실행 가능성을 먼저 검토한다. 멀티 인스턴스 환경이면 분산 락 또는 실행 제어 전략이 필수다.
- 실패 시 무한 재시도처럼 보이는 구조를 만들지 않는다.
- 외부 API 호출, 대량 데이터 처리, 오래 걸리는 작업은 작은 단위로 나누고 실행 시간을 통제한다.

## 1) 스케줄러를 사용하는 경우

- 정해진 시간/주기마다 상태를 동기화해야 할 때
- 만료 처리, 정산, 집계, 알림 발행처럼 반복 백그라운드 작업이 필요할 때
- 사람이 직접 호출하지 않아도 되는 운영성 작업일 때
- 이벤트 기반 처리로 풀기 어려운 주기성 업무일 때

## 2) 스케줄러를 사용하지 않는 경우

- 즉시성이 중요한 사용자 요청 처리
- 단순히 “나중에 한번 실행”만 필요한 일회성 작업
- 메시지 큐/이벤트 소비 구조가 더 적합한 경우
- 실행 시간이 길고 재시도/복구/모니터링이 중요한데 스케줄러만으로 감당하기 어려운 경우

## 3) 구현 원칙

- `@Scheduled` 메서드는 얇게 유지한다.
- 스케줄러는 “언제 실행할지”만 담당하고, “무엇을 할지”는 서비스로 위임한다.
- 한 번에 너무 많은 데이터를 잡지 말고 chunk 단위로 처리한다.
- 재실행돼도 안전한 idempotent 구조를 우선한다.
- 로그, 실행 시작/종료, 처리 건수, 실패 건수는 반드시 남긴다.

## 4) 트랜잭션 전략

- 긴 작업 전체를 하나의 큰 트랜잭션으로 묶지 않는다.
- 읽기와 쓰기, 외부 API 호출과 DB 저장 단계를 분리한다.
- chunk 단위 또는 대상 단위로 짧은 트랜잭션을 사용한다.
- 외부 API 호출을 트랜잭션 안에서 오래 붙잡지 않는다.

## 5) 중복 실행 방지

- 단일 인스턴스라고 가정하지 않는다.
- 운영 환경이 멀티 인스턴스면 분산 락 또는 단일 실행 보장 장치가 필요하다.
- 실행 중 플래그만으로 중복 실행을 막는 방식은 인스턴스 간 보장이 안 되므로 기본 전략으로 삼지 않는다.
- 작업은 가능하면 중복 실행돼도 안전하도록 설계한다.

## 6) 외부 API/대량 처리 작업

- 외부 API 호출은 timeout, retry, 실패 로그 전략을 함께 둔다.
- 대량 대상 작업은 paging/chunking으로 나눈다.
- 실패 항목과 성공 항목을 분리 기록해 전체 롤백을 피한다.
- 필요하면 스케줄러는 대상만 수집하고, 실제 처리 로직은 별도 작업 단위로 위임한다.

## 7) Coroutine과 함께 쓸 때 기준

- 스케줄러 자체를 Coroutine 기반으로 바꾸는 것이 기본 전략은 아니다.
- 외부 API 여러 개를 병렬 호출해야 하는 특정 작업에서만 제한적으로 검토한다.
- JPA 중심 배치 작업이면 동기 방식 유지가 기본이다.
- Coroutine을 쓰더라도 트랜잭션 경계와 blocking I/O 경계는 명확히 분리한다.

## 8) 구현 전에 빠르게 확인할 질문

- 이 작업은 정말 주기 실행이 필요한가?
- 이벤트 기반 처리보다 스케줄러가 더 적합한가?
- 중복 실행돼도 안전한가?
- 멀티 인스턴스에서 한 번만 돌아야 하는가?
- 실행 시간이 길어질 때 chunking 전략이 있는가?
- 실패/재시도/알림/로그 정책이 있는가?

## Completion Gate

- `@Scheduled` 메서드가 얇고 역할이 분리됐는가?
- 중복 실행 방지 전략이 설계에 포함됐는가?
- 긴 트랜잭션과 외부 API 호출이 분리됐는가?
- 운영 시 추적 가능한 로그와 실패 정보가 남는가?
