# Patterns

## Purpose

이 스킬은 도메인 로직 외부에서 반복되는 **아키텍처 패턴**을 도입하거나 설계할 때 사용한다.
"이 패턴을 써야 하는가"와 "어떻게 구현하는가"를 함께 다룬다.

## References

- 스케줄러 도입 기준 및 구현 원칙: `references/scheduler.md`
- Kotlin Coroutine 도입 기준 및 허용 범위: `references/coroutine.md`
- 이벤트 발행/리스너 패턴: `references/event.md`

## 패턴 선택 기준 요약

| 패턴 | 도입 기준 |
|---|---|
| Spring Scheduler | 정해진 주기로 반복되는 백그라운드 작업 |
| Coroutine | 외부 I/O 병렬 호출, 대기 시간이 긴 I/O, 스트리밍 응답 |
| 이벤트 발행 | 사이드이펙트를 메인 트랜잭션에서 분리할 때 |

## 공통 원칙

- 먼저 "이 패턴 없이 해결 가능한가"를 먼저 질문한다.
- 복잡도가 이점보다 크면 단순한 동기 방식을 선택한다.
- 패턴을 도입하면 실패/재시도/타임아웃/로그 정책을 함께 설계한다.
