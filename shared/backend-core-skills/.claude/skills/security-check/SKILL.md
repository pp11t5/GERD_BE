# Security Check

## Purpose

이 스킬은 기능 구현 전 self-check, PR 리뷰, 보안 회귀 테스트 설계에서 보안 누락을 줄이기 위해 사용한다.
프로젝트 고유 기술보다 인증, 인가, 입력 경계, 민감정보, 상태 전이, 외부 연동 같은 공통 보안 관점을 우선 점검한다.

## Critical Rules

- 인증과 인가를 구분해서 검토한다.
- 사용자 입력은 위치와 형식에 상관없이 신뢰하지 않는다.
- 민감정보는 로그, 응답, 예외, 저장소에 불필요하게 남기지 않는다.
- 공개 기능과 보호 기능의 경계를 문서와 설정 양쪽에서 확인한다.
- 보안 이슈는 편의보다 기본 차단을 우선한다.

## 1) 인증과 인가

- 인증이 필요한 엔드포인트가 공개 설정으로 열려 있지 않은지 먼저 확인한다.
- 인증 주체는 프로젝트 표준 방식으로 주입받고, null 가능성과 비정상 인증 상태를 함께 점검한다.
- 인증 성공 여부만 보지 말고 리소스 소유권, 역할, 소속, 권한 범위 같은 인가 조건까지 검증한다.
- 컨트롤러뿐 아니라 서비스 계층에서도 요청자가 접근 가능한 대상인지 다시 검증한다.
- 식별자만 바꾸면 다른 사용자 데이터가 조회/수정되는 IDOR 패턴이 없는지 확인한다.
- 관리자 또는 운영자 기능이 일반 사용자 권한으로 우회되지 않는지 테스트한다.

## 2) 입력 검증과 요청 경계

- DTO 또는 요청 모델에 기본 입력 검증을 적용하고, 비즈니스 규칙은 서비스에서 검증한다.
- enum, 정렬 조건, 페이지 크기, 커서, 상태값은 허용 범위를 명시한다.
- path variable, query parameter, header, request body 모두 신뢰하지 않는다.
- 업로드 요청은 content type, MIME, 확장자, 저장 경로 규칙을 함께 검증한다.
- 검색, 정렬, 필터 조건은 화이트리스트 기반으로 제한한다.
- nullable 입력이나 기본값 처리가 권한 우회, NPE, 예외 누락으로 이어지지 않는지 확인한다.

## 3) 개인정보와 민감정보

- 개인정보, 인증정보, 토큰, 코드, 키 값은 로그에 남기지 않는다.
- 예외 메시지에 내부 식별자, SQL, 외부 API 원문, 스택 세부정보를 노출하지 않는다.
- 응답 DTO에 불필요한 내부 상태, 관리자 전용 필드, 시스템용 식별자를 포함하지 않는다.
- 외부 접근 URL이나 저장소 key는 필요한 범위에서만 노출한다.
- DB, 캐시, 메시지, 이벤트 payload에 민감정보를 저장한다면 저장 이유, 보관 기간, 삭제 정책을 설명 가능해야 한다.

## 4) 캐시와 임시 저장소

- 키 생성 규칙은 중앙화하고 문자열 조합을 여러 곳에 흩뿌리지 않는다.
- 키와 값에 평문 개인정보를 직접 저장하지 않는다.
- 저엔트로피 값은 해시만으로 보호하지 말고 TTL, 시도 횟수 제한, 삭제 정책을 함께 둔다.
- 캐시 miss나 저장소 장애 시 fallback 경로를 설계하고 테스트한다.
- TTL, invalidation, 재계산 조건을 함께 검토한다.
- 멀티 스텝 업데이트는 부분 실패 시 데이터 불일치가 남는지 확인한다.

## 5) 토큰, 코드, 세션성 데이터

- 인증 코드와 일회성 토큰은 성공 시 즉시 무효화하거나 삭제한다.
- 실패 횟수 제한과 TTL을 함께 둔다.
- 재발급, 재설정, 인증 흐름은 재사용 공격이 불가능해야 한다.
- 토큰 비교 시 null 처리, 만료 여부, 상태 검증을 먼저 수행한다.
- 토큰 저장 방식은 평문 여부, 저장 위치, 만료 정책을 명확히 한다.
- 관련 API는 존재 여부를 과도하게 노출하지 않도록 응답 메시지를 점검한다.

## 6) API 설계와 에러 응답

- 민감한 값 전달은 가능하면 URL보다 요청 본문이나 보안이 강화된 방식으로 처리한다.
- 검증 실패, 권한 실패, 리소스 없음, 상태 충돌은 구분된 에러 코드로 표현한다.
- 에러 응답은 프로젝트 공통 포맷을 따른다.
- 500 응답에 내부 스택, 클래스명, 외부 서비스 세부 오류를 포함하지 않는다.
- 공개/비공개 엔드포인트 설정은 문서와 실제 보안 설정이 일치해야 한다.
- CORS, CSRF, 공개 URL, 업로드 정책 같은 엣지 설정은 변경 시 영향 범위를 기록한다.

## 7) 업로드와 외부 연동

- 업로드/다운로드 URL은 만료시간과 접근 범위가 최소 권한인지 확인한다.
- 업로드 경로는 사용자별 또는 정책상 허용된 prefix만 사용한다.
- 외부 API 호출 실패 시 민감정보가 포함된 로그를 남기지 않는다.
- 웹훅, 알림, 메일 전송은 식별자 노출 없이 추적 가능한 최소 로그만 남긴다.
- 외부 연동 응답을 그대로 클라이언트에 전달하지 않는다.

## 8) 동시성, 상태 전이, 데이터 무결성

- 정원 제한, 중복 생성 방지, 한도 차감은 락, 유니크 제약, 원자적 갱신 등으로 보호한다.
- count 후 save 같은 패턴은 레이스 컨디션이 없는지 확인한다.
- 상태 전이 API는 허용된 이전 상태에서만 호출 가능한지 검증한다.
- 삭제 후 재생성, 중복 요청, 재시도 시 동일성 보장이 필요한지 확인한다.
- 이벤트 발행이 있는 쓰기 로직은 트랜잭션 경계와 발행 시점을 검토한다.

## 9) 테스트 체크

- 인증 성공, 비인증 실패, 권한 실패를 모두 테스트한다.
- status만 보지 말고 에러 코드와 응답 구조까지 검증한다.
- IDOR 가능성이 있는 API는 타 사용자 시나리오를 추가한다.
- 업로드 API는 허용되지 않은 content type, MIME, 경로 접근을 실패 케이스로 검증한다.
- fallback, 토큰 재사용 방지, 실패 횟수 제한은 회귀 테스트로 남긴다.
- 문서화 대상 API는 실제 보안 정책과 문서가 일치하는지 확인한다.

## 10) PR 리뷰용 빠른 질문

- 이 변경으로 다른 사용자의 데이터를 조회하거나 수정할 수 있는 경로가 생기지 않는가?
- 로그, 예외, 응답에 민감정보가 새로 노출되지 않는가?
- 캐시, 저장소, 이벤트 payload에 평문 민감정보가 들어가지 않는가?
- 인증만 있고 인가 검증이 빠진 부분은 없는가?
- 실패 시나리오가 테스트와 에러 코드로 남아 있는가?

## Completion Gate

- 인증/인가 경계가 코드와 테스트로 보장되는가?
- 민감정보 노출 가능성이 제거됐는가?
- 토큰, 캐시, 업로드, 외부 연동의 실패 경로까지 점검됐는가?
- 보안 실패 시 응답이 과도한 정보를 주지 않는가?
