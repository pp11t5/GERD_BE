# Refactoring

## Purpose

이 스킬은 동작을 유지한 채 구조를 개선할 때 사용한다.
한 번에 크게 바꾸기보다 검증 가능한 작은 단위로 나눈다.
리팩터링의 핵심 목표는 가독성, 유지보수성, 중복 제거다.

## Critical Rules

- 기능 변경과 리팩터링을 한 커밋/한 변경에 과도하게 섞지 않는다.
- 보호 테스트 없이 큰 구조 변경을 하지 않는다.
- 이름을 바꿀 때는 의미 개선이 명확해야 한다.
- 공통화는 중복 2줄이 아니라 반복되는 개념이 있을 때만 한다.
- 가독성, 유지보수성, 중복 제거를 우선하고 미세 최적화는 뒤로 미룬다.
- 줄 간격, 정렬, 괄호 스타일 같은 형식 문제는 리팩터링의 핵심으로 보지 않는다.
- Controller에서 비즈니스 로직을 걷어내고 Service에 책임을 모은다.
- Service에 HTTP/웹 타입이 스며들지 않게 유지한다.
- DTO에 엔티티 생성 로직을 넣는 식의 책임 이동은 리팩터링으로 보지 않는다.
- 문제 재현 없이 추측으로 구조를 바꾸지 않는다.
- AI를 활용하더라도 초안 생성 후 개발자 검토와 테스트를 반드시 거친다.

## What Matters

- 가독성
  - 이름이 의미를 드러내는가
  - 함수 역할이 분명한가
  - 로직 흐름이 단순한가

- 유지보수성
  - 수정이 쉬운 구조인가
  - 확장 시 영향 범위가 작아졌는가
  - 테스트 가능한 경계가 생겼는가

- 중복 제거
  - 같은 규칙이나 변환이 여러 곳에 흩어져 있지 않은가
  - 한 번 수정하면 여러 곳이 같이 정리되는가

## What Does Not Matter First

- 근거 없는 미세 성능 최적화
- 포맷터가 해결할 스타일 집착
- 추상화만 늘리는 구조 미화

## Workflow

1. 개선이 필요한 문제를 파악한다. 중복 코드, 지나치게 긴 메서드, 책임 누수, 잘못된 의존성
2. 현재 동작을 고정할 테스트를 확보한다.
3. 리팩터링 목표를 가독성, 유지보수성, 중복 제거 중 어디에 두는지 먼저 적는다.
4. 작은 단계로 이동/분리/추출/이름 변경을 수행한다.
5. 각 단계마다 컴파일/테스트를 확인한다.
6. 최종적으로 책임 경계와 네이밍을 점검한다.

## AI Use Rule

- AI에게는 “기능 변경 금지”, “가독성 개선”, “중복 제거”, “책임 분리”, “출력 형식”을 명확히 준다.
- AI 결과는 초안으로 보고 바로 반영하지 않는다.
- 최종 반영 전에는 반드시 코드 리뷰와 테스트를 거친다.

## Good Targets

- Controller의 엔티티 조립 제거
- DTO/Converter/Service 책임 혼재 해소
- Query 조건 메서드와 커서/정렬 로직 분리
- 한 메서드에 섞인 권한 검증, 상태 전이, 저장 책임 분리
- 반복되는 변환 로직의 converter 추출

## Do Not

- 테스트 없이 대규모 파일 이동
- 템플릿 원문을 그대로 도입한 뒤 리팩터링했다고 간주하기
- 한 번만 쓰이는 1~2줄 메서드를 무분별하게 추출하기
- 추상화만 늘리고 실제 중복/복잡도는 줄이지 못하는 변경

## Completion Gate

- 변경 전후 동작이 동일함을 설명할 수 있는가?
- 클래스/메서드 책임이 더 선명해졌는가?
- 추상화가 실제 중복을 줄였는가?
- 가독성, 유지보수성, 중복 제거 중 최소 한 가지 이상이 분명히 개선됐는가?
