# Service Layer

## Purpose

이 스킬은 현재 프로젝트의 코딩 룰에 맞춰 서비스 계층 책임, SOLID 적용 기준, 트랜잭션 경계, 예외 처리, 변환 책임을 정리할 때 사용한다.

## Critical Rules

- Service는 유즈케이스 단위로 동작해야 한다.
- 읽기/쓰기 책임이 다르면 분리를 우선 검토한다.
- 클래스 기본은 `@Transactional(readOnly = true)`를 둔다.
- 쓰기 메서드에는 메서드 단위 `@Transactional`을 명시한다.
- 외부 I/O와 핵심 트랜잭션을 가능한 한 분리한다.
- 도메인 검증은 Controller가 아니라 Service/Domain에서 보장한다.
- 비즈니스 규칙은 서비스에 배치한다.
- DTO는 스키마 역할에 집중시키고, 엔티티 조립/변환은 Service 또는 converter에서 처리한다.
- 반복되는 변환은 converter로 추출하고, 1회성 단순 매핑만 Service 내부에 남긴다.
- 서비스는 Swagger, `ApiResponse` 설명, 문서 책임을 갖지 않는다.
- 예외는 `CustomException` + 도메인 `*ErrorCode` 기준으로 맞추고 내부 스택/메시지는 외부에 노출하지 않는다.
- 메서드 분리는 읽기 쉬움과 변경 단위 분리가 목적일 때만 수행한다.
- 한 번만 쓰이는 1~2줄 래퍼 private 메서드를 남발하지 않는다.
- Kotlin 스타일은 불변 우선(`val`), `!!` 금지, 명시적 예외 사용을 기본으로 둔다.
- 템플릿 스타일처럼 조회 메서드는 `requireNotNull`, 도메인 예외, 응답 DTO 조립 흐름이 읽히게 유지한다.
- 의존성 주입은 필드 주입보다 생성자 주입을 기본으로 사용한다.
- nullable 반환은 정말 의미가 있을 때만 허용하고, 없으면 도메인 예외나 명시적 결과 타입으로 처리한다.
- 코루틴을 도입하더라도 Service의 유즈케이스 책임과 트랜잭션 경계가 흐려지지 않게 유지한다.

## Kotlin / Kofring Rules

- 생성자 주입을 기본으로 사용하고, 주 생성자에 의존성을 선언한다.
- `val`을 우선하고, mutable 상태는 꼭 필요한 범위로 제한한다.
- `!!` 대신 null 가능 타입, 안전 호출, 엘비스 연산자, 명시적 예외를 사용한다.
- Java `Optional`은 서비스 경계에서 바로 Kotlin nullable 또는 도메인 예외로 정리한다.
- 스코프 함수는 가독성 이득이 분명할 때만 사용한다.

## Nullable 처리 기준

- 조회 결과가 “없을 수 있음” 자체가 유즈케이스 의미라면 nullable 반환을 허용한다.
- 조회 결과가 반드시 있어야 다음 로직이 성립하면 Service에서 바로 예외로 전환한다.
- nullable 처리 책임을 Controller까지 불필요하게 넘기지 않는다.
- `requireNotNull`은 내부 불변식 확인에 사용하고, 외부 계약 실패는 도메인 예외로 처리한다.

## Coroutine 기준

- 동시성이 필요하거나 I/O 대기가 큰 구간에서만 코루틴을 검토한다.
- 단순 MVC/JPA 흐름을 코루틴으로 무리하게 바꾸지 않는다.
- `suspend` Service를 도입하면 호출 체인과 테스트 전략도 함께 맞춘다.
- 여러 외부 호출을 병렬화할 때만 `async/await`를 검토하고, 순차 유즈케이스는 평범한 suspend 흐름을 유지한다.
- 코루틴 도입 후에도 예외 처리, 트랜잭션 경계, 타임아웃이 읽히게 유지한다.

## SOLID 적용 기준

- SRP: 컨트롤러는 API 경계, 서비스는 비즈니스 규칙, 리포지토리는 영속성 책임으로 분리한다.
- OCP: 기존 서비스 흐름을 깨는 수정보다 전략 또는 분기 메서드 추가로 확장하는 쪽을 우선한다.
- LSP: 인터페이스나 추상 타입 계약을 깨는 예외적 반환(`null`, 타입 축소)을 두지 않는다.
- ISP: 과도한 범용 인터페이스 대신 도메인 목적별로 작게 분리한다.
- DIP: 서비스는 구현체 세부사항보다 인터페이스나 추상화에 의존하도록 유지한다.
- 단순 CRUD까지 과도하게 추상화하지 않고 실용적으로 적용한다.

## 트랜잭션 기준

- 읽기 중심 서비스는 클래스 레벨 `@Transactional(readOnly = true)`를 기본값으로 둔다.
- 생성, 수정, 삭제, 상태 전이 메서드에는 메서드 레벨 `@Transactional`을 사용한다.
- 카운트 제한, 정원 제한, 중복 생성 방지 같은 동시성 민감 로직은 잠금 대상 집계 엔티티를 같은 트랜잭션에서 조회하고 `count + save`를 처리한다.
- 동시성 제어가 필요한 경우 `PESSIMISTIC_WRITE` 또는 유니크 제약 중 더 적절한 방식을 먼저 선택한다.

## Split Criteria

- 단순 CRUD: 단일 Service 허용
- 상태 전이/검증 집중: CommandService 우선
- 검색/목록/통계/Projection: QueryService 우선
- 외부 연동과 재시도가 많음: Orchestrator 또는 별도 어댑터 검토

## Workflow

1. 입력과 출력 계약을 확인한다.
2. 상태 변경 여부를 기준으로 읽기/쓰기를 나눈다.
3. DTO에서 처리할지 converter로 뺄지 변환 책임을 먼저 정한다.
4. 읽기 메서드와 쓰기 메서드의 트랜잭션 선언을 정한다.
5. 동시성 민감 구간이면 잠금/제약/검증 순서를 결정한다.
6. Repository/외부 연동/이벤트 발행 순서를 설계한다.
7. 성공/실패 테스트를 추가한다.

## DTO vs Converter 판단 기준

- DTO에 둬도 되는 경우
  - 단순 필드 복사
  - 한 곳에서만 쓰는 응답 조립
  - 도메인 규칙이나 외부 조회가 섞이지 않음

- converter로 분리해야 하는 경우
  - 같은 변환이 여러 서비스/컨트롤러에서 반복됨
  - 여러 엔티티나 값 객체를 조합함
  - 상태/권한/포맷 규칙이 포함됨
  - 테스트를 독립적으로 붙이는 편이 더 명확함

## Completion Gate

- 트랜잭션 경계가 설명 가능한가?
- 읽기/쓰기 분리 기준이 일관적인가?
- 예외가 도메인 의미를 보존하는가?
- 테스트가 주요 실패 분기를 포함하는가?
- 클래스 기본 `readOnly = true`, 쓰기 메서드 `@Transactional` 원칙이 지켜지는가?
- 생성자 주입, nullable 처리, 코루틴 도입 이유가 설명 가능한가?
